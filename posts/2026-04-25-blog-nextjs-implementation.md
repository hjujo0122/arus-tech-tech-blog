[仕様駆動開発をはじめる記事](/posts/2026-04-27-1-spec-driven-dev-with-claude)を書きました。さっそく環境構築から始めます。
と言いましたが、その環境構築もAIに任せています。

自分でやったのは `C:\work` フォルダを作ってその中でClaude Codeを起動し「仕様駆動開発をするための準備をして」と言っただけです。

## 作られたもの

`C:\work` の中に以下の構成が作られました。

```
C:\work\
  CLAUDE.md              ← Claudeへの永続的な指示（自動生成）
  specs\
    templates\
      webapp.md          ← Webアプリ用テンプレート（自動生成）
  projects\
    <プロジェクト名>\
      specs\
      src\
```

全部自動生成です。フォルダを作ってClaudeを起動しただけの人間が何かをした顔をしています。

## CLAUDE.md がすべての起点

一番重要なのは `CLAUDE.md` です。これも自動生成です。

Claude Codeはプロジェクトのルートに `CLAUDE.md` があると、会話の開始時に自動で読み込みます。ここに「仕様書なしで実装しない」「仕様が変わったらコードより先に仕様書を更新する」というルールを書いておくことで、以降のすべての会話でClaudeが仕様駆動開発に従って動くようになります。

設定を人間が覚えなくていい、というのがポイントです。毎回「仕様書ファーストで進めてね」と言わなくても、`CLAUDE.md` がそれを担保してくれます。

実際に書かれている鉄則はこれです。

```
- 実装より先に仕様を書く。仕様のない機能は作らない。
- 仕様が変わったら、コードより先に仕様書を更新する。
- 曖昧な仕様は実装前にユーザーに確認する。
```

これがあるおかげで、「これ仕様に書いてないですが実装しますか？」と確認してくるようになります。

## テンプレート

`specs/templates/webapp.md` にWebアプリ用のテンプレートが作られました。

新しいプロジェクトを始めるときはこのテンプレートをコピーして `projects/<プロジェクト名>/specs/overview.md` に置き、中身を埋めてからClaudeに渡します。テンプレートがあると「何を書けばいいかわからない」という最初のハードルが下がります。

中身はこんな感じです。

| 項目 | 内容 |
|------|------|
| 目的 | 何のために作るか |
| 機能一覧 | Must / Should で優先度付き |
| スコープ外 | 作らないものを明示 |
| 画面一覧 | 画面遷移とURL |
| データモデル | 扱うデータの構造 |
| 技術スタック | 使う技術とその理由 |
| 受け入れ条件 | 完成の定義チェックリスト |

## README に使い方を書く

環境ができたので、別途「READMEに使い方を書いて」とお願いしました。

書いてもらったのは基本の流れです。

```mermaid
graph LR
  A[AIにSpecを作ってもらう] --> B[人がレビューする] --> C[AIが実装する]
```

各ステップのポイントは、**2のレビューで曖昧さを潰しきること**です。仕様が曖昧なまま実装に進むと、AIが合理的な選択をしてくれますが自分の意図とズレることがあります。レビューで「スコープが正しいか」「受け入れ条件が検証可能か」を確認してから渡すと、手戻りが減ります。
