Claude Codeを使っていると、同じことを何度も言い直す場面が出てきます。「記事はエンジニアじゃない人にもわかるように書いて」「このファイルを編集したらCRLF改行で保存して」──前の会話で話したことを次の会話でも覚えていてほしい、という話です。

Claude Codeにはそのための「メモリ機能」があります。最初はうまく設定できたと思っていたのですが、実際には動いていない場面があることに気づいて、改善しました。その一連の試行錯誤の話です。

## Claudeのメモリ機能とは

Claude Codeには、会話と会話のあいだをまたいで情報を記憶する仕組みがあります。仕組みはシンプルで、**ユーザーのホームフォルダにMarkdownファイルとして保存されているだけ**です。

保存先はこのあたりです（Windowsの場合）。

```
C:\Users\ユーザー名\.claude\projects\プロジェクト名\memory\
```

この中に `MEMORY.md` という目次ファイルがあり、そこから個別のメモファイルへのリンクが貼られています。

```markdown
# Memory Index

- [ブログ記事の書き方ガイド](blog_writing_guide.md) — 構成・文体・ルール
- [token-usage hookの設定](token_usage_hook.md) — 上限値・精度の限界
```

Claudeは会話を始めるとき、この `MEMORY.md` を読んで「この人はどんなことを覚えておいてほしいと思っているか」を把握します。詳細が必要なときは個別のファイルも読みに行きます。

メモリファイルは普通のテキストファイルなので、手動で編集することもできます。

## 何を記録させるか

メモリには4種類あります。

| 種類 | 何を書くか | 例 |
|------|-----------|-----|
| **user** | 自分のことや好み | 「Reactは初めてだがGoは10年の経験がある」 |
| **feedback** | 「こうしてほしい」という指示 | 「末尾にまとめを書くな」「テストにDBモックを使わない」 |
| **project** | 進行中の作業の背景・経緯 | 「このリファクタリングは法的要件が動機」 |
| **reference** | 外部リソースの場所 | 「バグはLinearの"INGEST"プロジェクトで管理」 |

今回使うのは主に **feedback** の種類です。

### 書くべきもの・書かないもの

ここが最初に勘違いしやすいところです。

**書くべきもの**は「次の会話でClaudeの行動が変わるもの」だけです。具体的には、「こうしてほしい」という指示や好み、作業の背景や動機などです。

**書かなくていいもの**は、コードやファイルを見ればわかるもの全般です。たとえば「スクリプトにこういう関数がある」「設定値は現在〇〇になっている」は、実際のファイルを読んだ方が確実です。メモに書いても、コードが変わったときに古い情報になるだけです。

最初は「今日やったことをまとめてメモしておこう」という発想で書いてしまいがちですが、それは作業ログであってメモリではありません。「この情報があると次の会話でClaudeがどう変わるか」を考えて絞るのが大事です。

## 「指摘したら自動でメモを更新して」という設定

実際にやったことはシンプルです。

> 「何か指摘した場合、自動でmemoryも修正するようにしたい」

とClaudeに伝えると、Claudeが自分でメモファイルを作って保存してくれました。

```markdown
---
name: 指摘・フィードバック時のmemory更新ルール
type: feedback
---

ユーザーが何かを指摘した場合、その会話の中で必ず以下を行う。

1. 指摘内容をfeedback memoryに保存する
2. 関連するガイドファイルがあれば同時に更新する
3. 後回しにしない

**Why:** ユーザーが明示的に要求した。
```

これ以降、指摘のたびにClaudeがメモを更新してくれる…はずでした。

## 動いていなかった

しばらく使っていると、指摘したのにメモが更新されていない場面が出てきました。

原因は仕組みの問題です。このメモリファイルはあくまで「Claudeが参考にする情報」として読み込まれるものです。作業に集中しているときに見落とすことがあります。Claudeにとって「やるべきこと」として認識させるには弱かった、ということです。

## より確実な方法：CLAUDE.mdに書く

Claude Codeには `CLAUDE.md` というファイルがあります。プロジェクトのルートに置くと、毎回必ず読み込まれる「強制力のある指示書」として機能します。メモリファイルとは違い、「参考にする」ではなく「必ず従う」扱いになります。

ただし `CLAUDE.md` に直接書くと、プロジェクトの開発ルールなど他の内容と混在して読みにくくなります。そこで別ファイル `MEMORY_RULES.md` に切り出して、`CLAUDE.md` からそこを参照する形にしました。

```markdown
# CLAUDE.md の一部

@MEMORY_RULES.md

# 仕様駆動開発ガイド
...
```

`MEMORY_RULES.md` の中身はこんな感じです。

```markdown
# Memory更新ルール

指摘・フィードバックを受けたら**同じターンで**必ず更新する。後回しにしない。

| 指摘の種類 | 更新先 |
|-----------|--------|
| 書き方・文体 | writing-guide.md |
| 開発フロー | 該当specsファイル |
| Claudeの動き方 | このファイル＋CLAUDE.md |
| その他 | 適切なmemoryファイル |
```

## ファイルサイズに気をつける

`CLAUDE.md` や `MEMORY_RULES.md` は毎回の会話で読み込まれます。つまりファイルが長くなるほど、毎回のトークン消費が増えます。

なのでこれらのファイルはできるだけコンパクトに保つ必要があります。追記するときは既存の内容を見直して冗長な部分を削る、古くなったルールは消す、という運用が大事です。「とりあえず書いておこう」を続けると、じわじわトークンを食いつぶしていきます。

## メモは「なぜ」まで書く

ルールを書くときに大事なのは、指示の内容だけでなく**理由も一緒に記録すること**です。

たとえば「テストにDBモックを使わない」という指摘だけだと、Claudeは「すべてのテストでDBモックを禁止」と解釈するかもしれません。「本番DBとモックの差異で過去に障害が起きたから」という理由があれば、例外的なケースで適切に判断できます。

```markdown
テストにDBモックを使わない。

**Why:** 本番DBとモックの差異で過去に障害が起きたため。
**How to apply:** 統合テスト全般に適用。単体テストは除く。
```

理由があると、想定外の状況でも判断できます。理由のないルールは、Claudeが「これって今のケースにも当てはまる？」と迷ったときに役立ちません。

## 限界

完璧には動きません。`CLAUDE.md` に書いても、複雑な作業中に見落とすことはあります。大事な指摘は「これメモしておいて」と明示的に言った方が確実です。

ただ、ゼロよりはずっとましです。「また同じこと言わなきゃいけないのか…」というストレスは減りました。完璧に動くシステムを目指すより、「だいたい動く」を維持しながら運用でカバーする、くらいの温度感がちょうどいいと思います。
